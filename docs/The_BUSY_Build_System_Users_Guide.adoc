== Introduction

BUSY (for _BU_ild _SY_stem) is a lean, cross-platform build system for the GCC, Clang and MSVC toolchains. It requires only a C89 compiler to bootstrap and produces a single small binary. Compared to CMake, QMake, Meson or GN, BUSY is characterized by a statically typed build specification language and the ability to build a project directly from scratch without any further requirements on the host system.

BUSY is based on and integrated with the Lua virtual machine (written in C89, not in Lua). Lua is one of the easiest code bases to build on all platforms; the only requirement is a C89 compiler. BUSY follows this tradition and benefits from the great work of the Lua authors.

This guide explains how to add BUSY to a project and how to build projects using BUSY. The companion document, _The BUSY Build System - Language and Built-ins Specification_, serves as the complete language reference.

=== Key Features

* *Statically typed* build specification language with strong type checking
* *Minimal requirements*: only a C89 compiler needed to bootstrap
* *Cross-platform*: supports Linux (x86, x86_64, ARM), Windows (MSVC, MinGW), and macOS (Clang)
* *Direct builds*: can run builds directly without Make, NMake, or Ninja
* *Modular*: explicit module hierarchy with information hiding and visibility control
* *Parameterizable*: submodules can accept parameters, enabling flexible feature configuration
* *Lean*: the entire build system compiles in seconds from source

=== How BUSY Works

BUSY conceptually runs in three phases, similar to Bazel:

. *Loading phase*: parses all BUSY files in the source tree
. *Analysis phase*: runs the statements and creates work trees for the selected products
. *Execution phase*: runs the selected work tree depth-first, invoking the compiler and linker

Even if all declarations are considered and different work trees are created, only the ones selected as default (identifier marked with `!`) or by the `-T` command line option are actually executed.

[[getting-started]]
== Getting Started

=== Prerequisites

You need a C89-compatible compiler:

* *Linux*: GCC or Clang (typically pre-installed, or install via `apt install gcc` or similar)
* *Windows*: MSVC (Visual Studio Build Tools) or MinGW
* *macOS*: Clang (install via Xcode Command Line Tools: `xcode-select --install`)

No other dependencies are required. BUSY does not need Python, CMake, Ninja, Make, or any other build tool.

[[building-busy]]
=== Building BUSY from Source

Building BUSY is remarkably simple:

. Clone or download the BUSY source code from https://github.com/rochus-keller/BUSY[github.com/.../BUSY]

. Open a terminal in the BUSY source directory

. Compile all source files with a single command:

On Linux or macOS:

....
cc *.c -O2 -lm -o lua
....

On Windows with MSVC:

....
cl /O2 /MD /Fe:lua.exe *.c
....

The result is a Lua executable with BUSY integrated. The compilation takes only a few seconds.

=== Project Directory Layout

The typical way to use BUSY is to place the BUSY source code in a build subdirectory within your project's source tree. A minimal project looks like this:

....
my_project/
    BUSY                  # root BUSY file (build specification)
    main.c                # your source code
    build/                # build directory
        (BUSY source files: *.c, *.h, build.lua, builtins.lua)
        output/           # generated by BUSY at build time
....

Alternatively, you can keep the BUSY source code at a separate location and use command line options to specify the source and build directories. The important thing is that your project's root directory contains a file named `BUSY` (or `BUSY.busy`).

== Your First BUSY Project

[[hello-world]]
=== A Simple "Hello World"

Let us build a simple C program. Create a project directory with two files:

*main.c*:

....
#include <stdio.h>
int main() {
    printf("Hello from BUSY!\n");
    return 0;
}
....

*BUSY* (the build specification file):

....
let hello ! : Executable {
    .sources += [ ./main.c ]
}
....

This BUSY file declares:

* `let` -- an immutable variable declaration
* `hello` -- the variable name (which becomes the default executable name)
* `!` -- the _default_ visibility mark, meaning this product is built by default
* `: Executable` -- the type is `Executable`, a predeclared class for building binaries
* `{ ... }` -- a constructor block that configures the object
* `.sources += [ ./main.c ]` -- adds `main.c` (relative to this BUSY file) to the source list

[[running-the-build]]
=== Running the Build

To build the project, navigate to the directory containing the compiled BUSY (the `lua` binary and `build.lua` script), then run:

....
lua build.lua <path-to-source>
....

For example, if you placed the BUSY source code in a `build` subdirectory:

....
cd my_project/build
cc *.c -O2 -lm -o lua
lua build.lua ..
....

BUSY will:

. Parse the root BUSY file (and any submodules)
. Analyze the declarations and build a work tree
. Compile `main.c` and link it into an executable named `hello`

The output appears in the `output` directory (by default `./output` relative to where you run `lua`).

=== Understanding the Output

After a successful build, you will find the compiled executable in the `output` directory. The exact subdirectory structure is implementation-specific, but you can use the `Copy` class (described in <<copy-targets>>) to place outputs at known locations.

== Tutorial: Building a Multi-File C Project

This section walks through building a progressively more complex project, introducing BUSY concepts along the way.

[[multi-file-project]]
=== Project Structure

Consider a project with a library and an executable:

....
calculator/
    BUSY
    main.c
    mathlib/
        BUSY
        calc.c
        calc.h
....

[[submodules]]
=== Submodules

The root `BUSY` file declares the `mathlib` subdirectory as a submodule:

....
submod mathlib

let app ! : Executable {
    .sources += [ ./main.c ]
    .deps += [ mathlib.sources ]
}
....

The `submod mathlib` declaration tells BUSY to look for a `BUSY` file in the `mathlib/` subdirectory. The `mathlib.sources` designator accesses the public variable `sources` declared in the mathlib module.

The `mathlib/BUSY` file:

....
let sources * : SourceSet {
    .sources += [ ./calc.c ]
    .include_dirs += [ . ]
}
....

Here:

* `sources` is the variable name
* `*` marks it as _public_, making it visible to outer and nested modules
* `SourceSet` is a collection of compiled sources that are not linked into a library; instead their object files are passed up to whatever depends on them
* `.include_dirs += [ . ]` adds the current directory to the include path so that `#include "calc.h"` works

[[configurations]]
=== Configurations

A `Config` object groups compiler flags, defines, include directories, and linker settings. This avoids scattering the same flags across many declarations.

....
let my_config : Config {
    .defines += [ "NDEBUG" ]
    .cflags_cc += "-std=c++11"
    if target_os == `win32 {
        .lib_names += [ "Ws2_32" "Advapi32" ]
    }
}

let app ! : Executable {
    .sources += [ ./main.cpp ]
    .configs += my_config
}
....

Configs are additive: values from configs are appended to those set directly on the product. The order matters -- values set directly on the product appear first, followed by configs in the order they are listed.

Configs can also be _composed_: a config can reference other configs via its own `.configs` field, creating reusable building blocks.

[[building-libraries]]
=== Building Libraries

The `Library` class produces `.a` / `.lib` (static) or `.so` / `.dll` / `.dylib` (shared) files.

....
let my_lib * : Library {
    .name = "mylib"
    .lib_type = `static
    .sources += [ ./foo.c ./bar.c ]
}
....

The `.lib_type` field accepts a `LibraryType` enumeration value: `` `static ``, `` `shared ``, or `` `framework ``.

The `.name` field is optional; if omitted, the variable name is used as the output file name.

For a shared library:

....
let my_shared_lib * : Library {
    .name = "mylib"
    .lib_type = `shared
    .sources += [ ./foo.c ./bar.c ]
    .configs += my_config
}
....

[[source-sets]]
=== Source Sets

A `SourceSet` compiles sources into object files but does not produce a library. The object files are implicitly passed up to whatever target depends on the source set. This is often preferable to static libraries:

* Faster builds (no intermediate library creation step)
* No dead code elimination issues with exported symbols

....
let core_sources * : SourceSet {
    .sources += [ ./core.c ./utils.c ]
    .defines += [ "CORE_MODULE" ]
}

let app ! : Executable {
    .deps += [ core_sources ]
    .sources += [ ./main.c ]
}
....

[[groups]]
=== Groups

A `Group` is a meta-target that collects dependencies without contributing any sources of its own. It simply passes through all results from its dependencies.

....
let all ! : Group {
    .deps = [ app tests tools ]
}
....

This is useful for defining a default build target that encompasses multiple products.

[[visibility]]
=== Visibility Marks

BUSY uses three visibility marks on declarations:

|===
|Mark |Name |Visible to

|`*`
|public
|outer and nested modules

|`-`
|protected
|nested modules only

|`!`
|default
|like public, plus marks the product as a default build target
|===

Declarations without a mark are private to the current module.

....
let internal_config : Config { ... }      # private
let shared_config * : Config { ... }      # public
let impl_detail - : SourceSet { ... }     # protected (visible to submodules)
let app ! : Executable { ... }            # default (public + auto-built)
....

[[parameters]]
=== Parameters

Parameters allow submodules to be configured by their parent modules. A parameter is declared with the `param` keyword:

....
# In a submodule's BUSY file:
param HAVE_THREADS = false
param HAVE_GUI = false

if HAVE_THREADS {
    # include threading support
}
....

The parent module passes values when declaring the submodule:

....
submod mylib = ./mylib (HAVE_THREADS, HAVE_GUI)
....

Passing just the name (e.g. `HAVE_THREADS`) is equivalent to `HAVE_THREADS = true`. You can also pass explicit values:

....
submod mylib = ./mylib (HAVE_THREADS = true, HAVE_GUI = false)
....

Parameters can only be of basic or enumeration type. They can also be set via the command line using the `-P` option:

....
lua build.lua .. -P HAVE_THREADS
lua build.lua .. -P "target_toolchain_path=//usr/local/bin"
....

[[conditional-compilation]]
=== Conditional Compilation

The `if` / `else` construct controls which declarations and statements are active. The predeclared variables `target_os`, `target_toolchain`, `build_mode` etc. are commonly used as conditions:

....
let config : Config {
    if target_os == `linux {
        .lib_names += [ "pthread" "dl" ]
        .ldflags += "-no-pie"
    } else if target_os == `win32 {
        .lib_names += [ "Ws2_32" "Advapi32" ]
    } else if target_os == `macos {
        .frameworks += [ "Foundation" "AppKit" ]
    }

    if target_toolchain == `gcc {
        .cflags_cc += "-std=c++11"
    } else if target_toolchain == `msvc {
        .cflags_cc += [ "/EHsc" "/GR" ]
    }

    if build_mode == `debug {
        .defines += [ "DEBUG" "_DEBUG" ]
    } else {
        .defines += [ "NDEBUG" ]
    }
}
....

[[outer-module-access]]
=== The Outer Module Operator

A nested module can reference declarations from its parent module using the `^` prefix:

....
# In the parent module:
let main_config - : Config {
    .include_dirs += [ ./include ]
}

submod mylib

# In mylib/BUSY:
let sources * : SourceSet {
    .sources += [ ./impl.c ]
    .configs += ^main_config    # references the parent's config
}
....

The `^` operator accesses the scope of the outer module. The referenced declaration must be declared _before_ the `submod` declaration, and must be visible (public or protected).

[[alternative-module-paths]]
=== Alternative Module Paths and Fallbacks

A `submod` declaration can reference any directory, not just a direct subdirectory:

....
submod qt = ../LeanQt (HAVE_GUI, HAVE_WIDGETS)
....

This declares a submodule named `qt` whose BUSY file is found at `../LeanQt/BUSY`, with the given parameters. The module hierarchy is _logical_ and does not need to mirror the file system hierarchy.

You can also specify a fallback file in case the submodule directory does not contain a BUSY file:

....
submod image else ./missing.busy
....

This is useful for optional components: if the `image` directory is deleted from the source tree, the `missing.busy` file is loaded instead. Typically, a missing file provides stub declarations that emit an error if actually used:

....
# missing.busy
let err : Message {
    .text = "this module was removed from the source tree"
}
let sources * : Group {
    .deps += err
}
....

== Tutorial: Building a Real-World Project

This section demonstrates BUSY usage with patterns drawn from real projects: https://github.com/rochus-keller/LeanQt[LeanQt], https://github.com/rochus-keller/leancreator[LeanCreator], https://github.com/rochus-keller/eigen[Eigen], and https://github.com/rochus-keller/oberon[Oberon+].

[[real-world-setup]]
=== Setting Up a Project with BUSY

The standard approach to adding BUSY to a project:

. Create a `BUSY` file in the root of your source tree
. For each subdirectory relevant to the build, create a `BUSY` file
. Add a `build` subdirectory and clone the BUSY source code into it (or place it alongside the source tree)
. Build the `lua` binary as described in <<building-busy>>

Here is a typical sequence to build any BUSY-managed project from scratch:

....
git clone https://github.com/user/project.git
cd project
mkdir build
cd build
git clone https://github.com/rochus-keller/BUSY.git .
cc *.c -O2 -lm -o lua
lua build.lua ..
....

NOTE: You can also keep the BUSY source code separate and use `-S` and `-B` to specify paths.

[[leanqt-example]]
=== Example: LeanQt

LeanQt demonstrates many advanced BUSY features. Its root BUSY file starts with parameter declarations that control which Qt features to include:

....
param HAVE_SHARED = false
param HAVE_FILEIO = false
param HAVE_COREAPP = false
param HAVE_OBJECT = false
param HAVE_GUI = false
param HAVE_WIDGETS = false
# ... many more HAVE_x parameters
....

Feature flags have automatic dependency resolution:

....
if HAVE_WIDGETS && !HAVE_GUI {
    message("enabling HAVE_GUI required by HAVE_WIDGETS")
    HAVE_GUI = true
}

if HAVE_GUI && !HAVE_IMAGE {
    message("enabling HAVE_IMAGE required by HAVE_GUI")
    HAVE_IMAGE = true
}
....

Submodules for each component are declared next:

....
submod core
submod mime else ./missing.busy
submod gui else ./missing.busy
submod widgets else ./missing.busy
submod tools
....

Note the use of `else ./missing.busy` for optional modules: if the directory is absent, a stub is loaded instead.

LeanQt provides both separate libraries and a combined "all-in-one" library:

....
# Separate libraries
let libqtcore * : Library {
    .deps += core_sources
    if HAVE_SHARED {
        .lib_type = `shared
        .configs += core_client_config
    } else {
        .lib_type = `static
    }
    .name = "qtcore"
}

# All-in-one library
let libqt * : Library {
    .deps += core_sources
    if HAVE_XML {
        .deps += xml_sources
    }
    if HAVE_GUI {
        .deps += gui_sources
    }
    # ... etc.
    if HAVE_SHARED {
        .lib_type = `shared
        .configs += qt_client_config
    } else {
        .lib_type = `static
    }
    .name = "qt"
}
....

To build LeanQt (e.g. from the Oberon+ project):

....
cd oberon
mkdir build && cd build
git clone https://github.com/rochus-keller/BUSY.git .
cc *.c -O2 -lm -o lua
lua build.lua ..
....

The Oberon+ root BUSY file imports LeanQt as a submodule:

....
submod qtmini = ../LeanQt (HAVE_FILEIO, HAVE_COREAPP, HAVE_PROCESS)
submod qtfull = ../LeanQt (HAVE_ITEMVIEWS, HAVE_PROCESS, HAVE_NET_MINIMUM)
....

This creates _two separate instances_ of the LeanQt module with different feature sets: a minimal one for the compiler and a full one for the IDE. Each `submod` declaration constitutes a separate module instance even when pointing to the same source directory.

[[moc-rcc-uic]]
=== Working with Qt Tools (moc, rcc, uic)

BUSY has built-in support for the Qt meta-object compiler (moc), resource compiler (rcc), and user interface compiler (uic). These are special action classes.

*Moc* processes header or source files with `Q_OBJECT` macros:

....
let my_moc : Moc {
    .tool_dir = root_build_dir + ^root_path
    .sources += [ ./mywidget.h ./mywidget.cpp ]
}

let my_app ! : Executable {
    .sources += [ ./main.cpp ./mywidget.cpp ]
    .deps += [ ^libqtcore ^libqtgui ^libqtwidgets my_moc ]
    .configs += [ ^core_client_config ^gui_client_config ^widgets_client_config ]
}
....

*Rcc* compiles `.qrc` resource files into C++ source:

....
let my_rcc : Rcc {
    .tool_dir = root_build_dir + ^root_path
    .sources += [ ./resources.qrc ]
}
....

*Uic* generates header files from `.ui` files:

....
let my_uic : Uic {
    .tool_dir = root_build_dir + ^root_path
    .sources += [ ./dialog.ui ]
}
....

When using these tools, add the action instance to the `.deps` of the product that needs the generated files. The generated sources are automatically compiled and linked.

[[leancreator-example]]
=== Example: LeanCreator

LeanCreator (a C/C++ IDE built with BUSY) demonstrates a large, modular project structure. Its root BUSY file:

....
submod qt - = ../LeanQt (HAVE_ITEMVIEWS, HAVE_PROCESS, HAVE_NETACCESS,
    HAVE_LOCALSOCK, HAVE_SHARED = HAVE_SHARED_QT,
    HAVE_SHAREDMEM, HAVE_XMLSTREAM, HAVE_CONCURRENT,
    HAVE_PLUGINS, HAVE_RE, HAVE_JSON, HAVE_XML, HAVE_ZLIB)

let cpp11conf - : Config {
    if (target_toolchain == `gcc) || (target_toolchain == `clang) {
        .cflags_cc += "-std=c++11"
    }
    .defines += "LEANCREATOR_ALL_STATIC"
}

submod utils -
submod extensionsystem -
submod core -
submod texteditor -
submod projectexplorer -
# ... many more submodules

submod app

let leancreator ! = app.exe
....

Notable patterns:

* LeanQt is declared with `-` visibility (protected) so only nested modules can access it
* Many submodules are also protected -- they are implementation details
* The final product is simply an alias: `let leancreator ! = app.exe`
* The `-` on the config (`cpp11conf -`) makes it available to all submodules but not to outside consumers

[[eigen-example]]
=== Example: Eigen (Multi-Architecture Compiler)

The Eigen project shows how to organize builds for multiple architectures from a shared code base:

....
let config * : Config {
    if target_toolchain == `gcc {
        .lib_names += [ "m" "stdc++" ]
        .ldflags += "-shared-libgcc"
        .cflags_cc += "-std=c++11"
        .cflags_c += "-std=c99"
    }
}

let common * : SourceSet {
    .configs += config
    .sources += [
        ./object.cpp ./driver.cpp ./code.cpp
        ./assembly.cpp ./asmgenerator.cpp
        # ... many more shared source files
    ]
}

let amd64 * : SourceSet {
    .configs += config
    .sources += [ ./amd64.cpp ./amd64assembler.cpp ./amd64generator.cpp ]
}

let cdamd64 : Executable {
    .configs += config
    .deps += [ common amd64 ]
    .sources += [ ./compiler.cpp ]
    .defines += [ "AMD64BACKEND" "CODEFRONTEND" "DEBUGGINGINFORMATION" ]
}
....

The pattern: a shared `SourceSet` (`common`) combined with architecture-specific source sets, each yielding different executables via `#define` flags.

[[copy-targets]]
=== Copy Targets

The `Copy` class copies files to known locations in the build tree. This is important because the internal structure of the build directory is implementation-specific.

....
let install ! : Copy {
    .use_deps += `executable
    .deps += [ my_app my_tool ]
    .outputs += root_build_dir + ./bin/{{source_file_part}}
}
....

The `{{source_file_part}}` placeholder is replaced with the file name of each copied file. Other placeholders include `{{source_name_part}}`, `{{source_ext}}`, `{{source_dir}}`, and `{{source}}`.

The `.use_deps` field specifies which file types to collect from dependencies. Possible values include `` `executable ``, `` `static_lib ``, `` `shared_lib ``, `` `source_file ``, and `` `object_file ``.

=== Scripts

BUSY supports running Lua scripts as part of the build:

*LuaScript* runs a script once:

....
let generate : LuaScript {
    .script = ./generate.lua
    .args += [ "{{root_build_dir}}" "output.h" ]
    .outputs += [ ./output.h ]
}
....

*LuaScriptForeach* runs a script once per source file:

....
let process : LuaScriptForeach {
    .script = ./process.lua
    .sources += [ ./input1.txt ./input2.txt ]
    .args += [ "{{source}}" "{{current_build_dir}}/{{source_name_part}}.out" ]
    .outputs += [ ./{{source_name_part}}.out ]
}
....

=== Macros

Macros combine a sequence of declarations and statements into a reusable command. They work at the lexical token level:

....
define add_test(name, src) {
    let name : Executable {
        .sources += [ src ]
        .deps += [ test_framework ]
        .configs += test_config
    }
}

add_test(test_math, ./test_math.cpp)
add_test(test_string, ./test_string.cpp)
....

Arguments are not evaluated at the call site; instead, tokens passed as arguments are inserted into the macro body wherever the argument is referenced. Identifiers can be concatenated using the `&` operator in the macro body.

=== Version Checking

BUSY files can check the BUSY version to ensure compatibility:

....
if busy_version < "2023-02-12" {
    error("this version of BUSY is not compatible with this build")
}
....

This is good practice for projects that use features introduced in a specific BUSY version.

=== The set_defaults Procedure

The `set_defaults` procedure sets a default configuration for a given toolchain. This is useful for controlling optimization or linking settings globally:

....
let mtconf : Config {
    .cflags = [ "/O2" "/MT" ]
}

if (build_mode == `optimized) && (target_toolchain == `msvc) {
    set_defaults(target_toolchain, mtconf)
}
....

Use this sparingly. Libraries should generally not set defaults. Prefer using `build_mode` to decide which defaults to apply.

[[alternative-syntax]]
=== The Alternative (Pascal) Syntax

BUSY supports an alternative syntax with a Pascal flair. Both syntax versions can be used interchangeably, even within the same project:

|===
|C-flair syntax |Pascal-flair syntax

|`let x = expr`
|`let x := expr`

|`x = expr`
|`x := expr`

|`if cond { ... } else { ... }`
|`if cond then ... elsif cond2 then ... else ... end`

|`let t : T { .x = 1 }`
|`let t : T begin .x := 1 end`

|`submod m = ./path`
|`submod m := ./path`
|===

For example, a `SourceSet` declaration in the alternative syntax:

....
let sources * : SourceSet
begin
    .sources := [
        ./draw2d.cpp
        ./drawg.cpp
        ./btext.c
    ]
    if target_os == `linux then
        .deps += gtk3.sources
    elsif target_os == `win32 then
        .deps += win.sources
    else
        error("target os not supported")
    end
end
....

== Command Line Reference

BUSY is invoked via the `build.lua` script:

....
lua build.lua [options] [<path-to-source>]
....

|===
|Option |Description

|`<path>` (no flag)
|Path to the root of the source tree (shortcut for `-S`)

|`-S <path>`
|Explicitly set the path to the source tree root (default: `..`)

|`-B <path>`
|Explicitly set the path to the build directory root (default: `./output`)

|`-T <target>`
|Select which product(s) to build; e.g. `-T my_lib`; can select nested products like `-T sub.my_lib`; can be specified multiple times

|`-P <key>=<value>`
|Set a parameter value; e.g. `-P HAVE_GUI` (implies `=true`) or `-P "target_toolchain_path=//usr/local/bin"`

|`-M <mode>`
|Set build mode: `optimized` (default), `nonoptimized`, or `debug`

|`-opt`
|Shortcut for `-M optimized`

|`-nopt`
|Shortcut for `-M nonoptimized`

|`-dbg`
|Shortcut for `-M debug`

|`-c`
|Check only: run the parser/analyzer without building; no files or directories are generated

|`-G <generator>`
|Generate project files for another build system; currently `-G qmake` is supported for QtCreator integration
|===

=== Examples

Build the default products from a parent directory:

....
lua build.lua ..
....

Build a specific target in debug mode:

....
lua build.lua .. -T compiler -M debug
....

Set parameters for cross-compilation:

....
lua build.lua ../LeanQt -P target_toolchain_path=//home/me/toolchain/bin \
    -P "target_toolchain_prefix=\"arm-linux-gnueabihf-\""
....

Check BUSY files without building:

....
lua build.lua .. -c
....

Generate QMake project files for use with QtCreator:

....
lua build.lua .. -G qmake
....

== Best Practices

=== Project Organization

* Place a `BUSY` file in the source tree root and in each relevant subdirectory
* Use `submod` declarations to make the module hierarchy explicit
* Keep the module hierarchy logical; it does not need to mirror the file system
* Use `else ./missing.busy` for optional components that may be absent

=== Visibility and Information Hiding

* Make only the declarations that consumers need *public* (`\*`) or *default* (`!`)
* Use *protected* (`-`) for declarations that should be accessible to submodules but not to the outside
* Keep internal implementation details private (no visibility mark)
* Configs shared between a library and its consumers should be public

=== Configuration Management

* Group related compiler flags, defines, and include paths into `Config` objects
* Provide a `client_config` for consumers of your library that includes the necessary public headers and defines
* Separate internal build configs from public-facing configs
* Use `set_defaults` sparingly and only in top-level application modules, not in libraries

=== Dependency Management

* Prefer `SourceSet` over `Library` for internal components -- it avoids the overhead of creating intermediate library files
* Use `Group` to collect related products under a single target name
* Keep the dependency tree explicit; cycles are not allowed
* Use parameters to enable or disable optional dependencies

=== Cross-Platform Builds

* Use `target_os` and `target_toolchain` for platform-specific settings
* Keep platform-specific code in separate source sets controlled by conditions
* Test on all target platforms early; BUSY makes this easy since it bootstraps anywhere

=== Naming Conventions

* Use descriptive names for variables: `core_sources`, `gui_config`, `libqt`
* Use the `.name` field on `Executable` and `Library` to control the output file name
* Avoid name collisions between executables and libraries in the same module

== Frequently Asked Questions

=== How do I add BUSY to an existing project?

. Create a `BUSY` file in the root of your source tree
. Declare your sources, products, and dependencies
. Create a `build` directory, clone the BUSY source into it, and compile
. Run `lua build.lua ..`

=== Can I use BUSY alongside CMake or other build systems?

Yes. BUSY files are independent of other build systems. You can keep your `CMakeLists.txt` or `Makefile` alongside the `BUSY` files. The BUSY file is simply named `BUSY` (no extension by default), so there are no file name conflicts.

=== How do I do cross-compilation?

Set the `target_toolchain_path` and optionally `target_toolchain_prefix` parameters:

....
lua build.lua .. -P target_toolchain_path=//path/to/cross/bin \
    -P "target_toolchain_prefix=\"arm-linux-gnueabihf-\""
....

The `cross_compiling()` function returns `true` when the host and target differ, which you can use in your BUSY files to conditionally adjust the build.

=== How do I debug my BUSY files?

* Use `-c` to run the parser/analyzer without building -- this catches type errors and missing declarations quickly
* Use the `dump()` procedure to print variable types and values
* Use `message()` to print diagnostic output during the analysis phase

=== What if a submodule directory is missing?

Use the `else` keyword in `submod` to specify a fallback:

....
submod optional_lib else ./missing.busy
....

The fallback file typically provides stub declarations. If someone tries to actually depend on the missing module, the stub can emit a build error via a `Message` with `` `error `` type.

=== How do I control the output location of my binaries?

Use the `Copy` class with `root_build_dir` to place outputs at known paths:

....
let install ! : Copy {
    .use_deps += `executable
    .deps += [ my_app ]
    .outputs += root_build_dir + ./bin/{{source_file_part}}
}
....
